    **Step 4: Create `src/app/[lang]/layout.tsx`**
    The locale-specific layout:
    - Import fonts from `@/lib/fonts` (marlinGeo, saprona, geistMono)
    - Import `NextIntlClientProvider`, `hasLocale` from `next-intl`
    - Import `getMessages`, `setRequestLocale` from `next-intl/server`

    - Import `notFound` from `next/navigation`
    - Import `routing` from `@/i18n/routing`

    `generateStaticParams()`: return `routing.locales.map((locale) => ({lang: locale}))`

    `generateMetadata()`: accept `params: Promise<{lang: string}>`, await params, get translations for `Metadata` namespace, return:
    - `title: t('title')`
    - `description: t('description')`
    - `alternates: { languages: { en: '/en/', vi: '/vi/' } }` -- this directly fixes FNDN-03 (hreflang)

    Default export async function:
    - Await `params` to get `lang`
    - Validate with `hasLocale(routing.locales, lang)`, call `notFound()` if invalid
    - Call `setRequestLocale(lang)`
    - Get messages via `await getMessages()`
    - Return `<html lang={lang}>` wrapping `<body className="{font variables} antialiased">` wrapping `<NextIntlClientProvider locale={lang} messages={messages}>{children}</NextIntlClientProvider>`

    IMPORTANT: The `<html>` and `<body>` tags go here, NOT in the root layout. The root layout should be a bare wrapper. Actually -- Next.js requires exactly one `<html>` and `<body>` per layout chain. So the root layout MUST have `<html>` and `<body>`, and the `[lang]` layout wraps content inside `<NextIntlClientProvider>` only, setting `lang` on the html via the root layout's pass-through.

    CORRECTION: The standard next-intl pattern for static export has the `<html lang>` in the `[lang]/layout.tsx`. But Next.js requires the root `layout.tsx` to also have `<html>` and `<body>`. The solution: root `layout.tsx` has `<html><body>{children}</body></html>` with no lang. The `[lang]/layout.tsx` does NOT re-declare html/body -- it just wraps children in `<NextIntlClientProvider>`. The `lang` attribute is set by having the root layout accept it from the nested layout somehow...

    Actually, the cleanest pattern for next-intl static export: Put `<html lang={lang}>` and `<body>` in `[lang]/layout.tsx`, and make the root `layout.tsx` simply `({children}) => children` (a passthrough). This is the documented next-intl pattern.

    So:
    - `src/app/layout.tsx`: `export default function RootLayout({children}) { return children; }` -- bare passthrough, no html/body tags
    - `src/app/[lang]/layout.tsx`: Contains `<html lang={lang}><body className="...">{provider wrapping children}</body></html>`
    - `src/app/page.tsx` and `src/app/not-found.tsx`: Need their own `<html><body>` wrappers since they're outside `[lang]`
